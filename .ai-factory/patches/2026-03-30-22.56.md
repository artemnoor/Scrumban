# Auth registration failed because Node runtime used the wrong database environment

**Date:** 2026-03-30 22:56
**Files:** apps/api/src/main.ts, apps/web/package.json, compose.yml, package.json, packages/config/package.json, packages/config/src/env-loader.ts, packages/config/src/index.ts, packages/db/package.json, packages/db/prisma/seed.ts, packages/db/scripts/prisma-command.ts, packages/db/src/client.ts, README.md, .env.example
**Severity:** high

## Problem

`POST /api/auth/register` returned `500 Internal Server Error` from the browser.
The frontend loaded, anonymous `/api/auth/me` checks returned `401` as expected, but
the first real database-backed auth request failed.

## Root Cause

There were two overlapping environment problems:

1. Node entrypoints did not load the repository `.env` file, so the API and DB scripts
   inherited whatever `DATABASE_URL` happened to be set in the current shell.
2. The machine already had another PostgreSQL process on `localhost:5432`, so the
   project's dev database could collide with a non-project instance.

Because of that combination, the app either targeted the wrong database or hit an
empty database with no schema, which surfaced as a server-side `500` during registration.

## Solution

Added a shared `loadNodeEnv()` helper that finds the nearest repo `.env`, loads it with
`override: true`, and emits `[FIX]` logs for visibility. Wired that helper into API and
database entrypoints so runtime and seed flows use project-local configuration.

Also added a Prisma wrapper script for DB commands, introduced `pnpm db:push`, and moved
the default Docker PostgreSQL host port to `5433` to avoid collisions with a locally
installed Postgres service on `5432`.

Finally, fixed a separate verification issue by adding the missing `zod` dependency to
`apps/web`, which restored `pnpm typecheck`.

## Prevention

- Load repo-local environment explicitly in Node entrypoints instead of assuming shell
  state is safe.
- Prefer non-default local Docker ports for common developer services like PostgreSQL.
- Keep a one-command schema bootstrap path such as `pnpm db:push` for first-run setups.
- Run `pnpm typecheck` after environment or package changes; it exposed the missing web
  dependency immediately.
