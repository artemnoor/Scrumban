# Local db push failed after switching tasks to board-scoped columns

**Date:** 2026-03-31 07:39
**Files:** package.json, packages/db/package.json, packages/db/scripts/prisma-command.ts, README.md
**Severity:** high

## Problem

`pnpm db:push` failed on a developer machine after the schema moved tasks from the old
status enum to the required `columnId` relation. Prisma reported a non-executable change
because the local `Task` table already contained rows and could not backfill the new
required column automatically.

After that, `pnpm db:seed` also failed because the new tables such as `BoardMember`
and `BoardColumn` were never created in the local database.

## Root Cause

The project quick-start path assumed a fresh database, but local development already had
an older schema with existing task rows. `prisma db push` is fine for bootstrapping a new
database, but it is not a safe replacement for an explicit migration strategy when a
required column is introduced on a populated table.

The tooling also did not give a clear next step, so the user saw the raw Prisma error
without a project-specific recovery command.

## Solution

Added explicit local recovery scripts:

- `pnpm db:reset` force-resets the local Prisma schema for development
- `pnpm db:bootstrap` runs generate + reset + seed in one pass

Also updated the Prisma command wrapper to emit a `[FIX]` hint when `db push` fails
without `--force-reset`, pointing the developer to `pnpm db:reset` followed by
`pnpm db:seed`.

README was updated so the documented setup path now distinguishes:
- fresh local schema: `pnpm db:push`
- old incompatible local schema: `pnpm db:reset`

## Prevention

- Do not document plain `db push` as the only setup path when the project is evolving
  across breaking schema changes.
- Give local dev environments an explicit reset/bootstrap command instead of forcing
  people to remember Prisma flags.
- When using `db push` in a monorepo, wrap it with project-aware guidance for common
  failure modes.
